  </read_first>

  <action>
    Create `.github/workflows/ci.yml` with the following structure:

    **Workflow triggers:**
    - Push to main, develop branches
    - Pull requests to main, develop branches

    **Jobs:**

    1. **backend job:**
       - Services: PostgreSQL 16 (port 5432), TimescaleDB (port 5433) with health checks
       - Go 1.22 setup with caching
       - Run tests with coverage: `go test -v -race -coverprofile=coverage.out -covermode=atomic ./...`
       - Generate coverage summary: `go tool cover -func=coverage.out`
       - Upload to Codecov: `codecov/codecov-action@v4` with `flags: backend`, `fail_ci_if_error: true`, `token: ${{ secrets.CODECOV_TOKEN }}`
       - Environment: DATABASE_URL, TIMESCALE_URL
       - Track test timing: use `-v` flag for verbose output showing per-test duration

    2. **frontend job:**
       - Node 18 setup
       - Install dependencies: `npm ci` in frontend directory
       - Run tests with coverage: `npm run test:coverage -- --run`
       - Upload to Codecov: `codecov/codecov-action@v4` with `flags: frontend`, `fail_ci_if_error: true`
       - Files: `./frontend/coverage/coverage-final.json`

    3. **lint job:**
       - Go 1.22 setup
       - golangci-lint: `golangci/golangci-lint-action@v3` with version `v2.11.4`, working-directory: backend, args: `--timeout=5m`
       - Node 18 setup
       - Frontend lint: `npm ci && npm run lint` in frontend directory

    4. **ci-passed job (summary for branch protection):**
       - depends on: [backend, frontend, lint]
       - `if: always()`
       - Check all job results, exit 1 if any failed
       - This single job becomes the required status check

    **Key patterns from research:**
    - Use `fail_ci_if_error: true` for Codecov uploads (ensures coverage failures block CI)
    - Use `codecov/codecov-action@v4` (not v3 - better token handling)
    - Coverage threshold checking happens via Codecov YAML (separate task), not in workflow
    - Test timing visible via `-v` flag output (Go) and verbose reporter (Vitest)

    **Do NOT:**
    - Do NOT set coverage threshold checks in the workflow itself (Codecov handles this)
    - Do NOT use `continue-on-error: true` for quality checks
    - Do NOT skip any existing test suites
  </action>

  <verify>

  </verify>

  <action>
    Create `codecov.yml` in repository root with:

    **Coverage configuration:**
    - precision: 2
    - round: down
    - range: "70...100"

    **Status checks:**
    - project.default:
      - target: 80% (ultimate goal)
      - threshold: 1% (allow 1% decrease without failing)
      - if_ci_failed: error
    - patch.default:
      - target: 80%
      - threshold: 5%

    **Comment layout:**
    - layout: "header, diff, files, footer"
    - behavior: default
    - require_changes: true

    **Flags:**
    - backend:
      - paths: ["backend/"]
      - carryforward: true
    - frontend:
      - paths: ["frontend/src/"]
      - carryforward: true

    **Ignore patterns:**
    - backend/cmd/** (main entry points)
    - backend/tests/** (test files)
    - frontend/src/test/** (test files)
    - frontend/e2e/** (E2E tests)
    - **/*.test.* (test files)
    - **/*.spec.* (spec files)

    **Critical note on thresholds:**
    Current backend coverage is ~9.1%. The 80% target is the ULTIMATE goal.
    Codecov will track progress and report coverage changes.
    Use `threshold: 1%` to allow gradual improvement without blocking every PR.

    **Do NOT:**
    - Do NOT set target below 80% (this is the milestone goal)
    - Do NOT exclude core source directories from coverage
    - Do NOT use `if_ci_failed: success` (should error on CI failure)
  </action>

  <verify>
