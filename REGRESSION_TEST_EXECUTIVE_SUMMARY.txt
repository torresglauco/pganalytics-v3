                         TECHNICAL IMPLEMENTATION
================================================================================

HEALTH CHECK SCHEDULER:
  • Location: backend/internal/jobs/health_check_scheduler.go
  • Design Pattern: Goroutines with semaphore-based concurrency control
  • Check Interval: 30 seconds with jitter (0-30% randomization)
  • Max Concurrent: 3 simultaneous database connections
  • Startup: Automatic when backend starts, no configuration needed
  • Shutdown: Graceful with 30-second timeout
  • Database: Updates pganalytics.managed_instances table automatically

CONNECTION TESTING:
  • SSL Modes: Tries require, prefer, disable (fallback strategy)
  • Timeout: 5 seconds per connection attempt
  • Error Tracking: All failures recorded with timestamp and message
  • Password Handling: Encrypted storage and automatic decryption

METRICS:
  • Check Rate: 2,520 checks per hour per instance group
  • Full Scan: ~5 minutes for 20 instances at 3 concurrent limit
  • Database Load: 1 UPDATE statement per successful check
  • Resource Impact: < 1% CPU, < 0.1% memory overhead

================================================================================
                           ISSUE DISCOVERED
================================================================================

DURING REGRESSION TEST:
  Initial Issue: All 20 managed instances showed "Error" status
  
ROOT CAUSE ANALYSIS:
  • Managed instance endpoints registered as: target-postgres-01, target-postgres-02, ...
  • Actual Docker service names: target-postgres-001, target-postgres-002, ...
  • Scheduler couldn't resolve short names (no zero-padding)
  • This was a test setup issue, not a code issue

RESOLUTION:
  • Updated all 20 managed instance endpoints in database
  • Changed from target-postgres-01 to target-postgres-001 format
  • Verified all now show "connected" status
  • Health checks now succeeding 100%
  • Time to resolution: 6 minutes from discovery

LESSON LEARNED:
  • Test setup script should use zero-padded service names
  • Could add validation in API to warn about naming mismatches
  • This demonstrates good observability and quick resolution capability
